---
title: "payloadtool"
description: "Validate and inspect payload generation from a lading configuration file."
---

`payloadtool` exercises the payload generation code paths from a lading configuration file without running a full experiment. Use it to verify that a payload config produces the expected output, measure memory consumption, and generate fingerprints for regression detection.

## Synopsis

```bash
payloadtool [OPTIONS] <CONFIG_PATH>
```

## Installation

`payloadtool` is built alongside `lading`:

```bash
cargo build --release --bin payloadtool
./target/release/payloadtool --help
```

## Arguments

<ParamField path="config_path" type="string" required>
  Path to a standard lading configuration YAML file. `payloadtool` reads the `generator` section and runs payload generation for each generator.
</ParamField>

## Options

<ParamField path="-g, --generator-id" type="string">
  Run payload generation for a single generator identified by its `id` field in the config. If omitted, all generators in the config are exercised.
</ParamField>

<ParamField path="-f, --fingerprint" type="boolean">
  Generate and print a fingerprint for the payload. The fingerprint is a SHA-256 hash combined with the Shannon entropy of the generated bytes, in the format `<sha256> entropy=<bits>`.

  Use fingerprints to detect unintended changes to payload generation.
</ParamField>

<ParamField path="-v, --verify" type="string">
  Path to a file containing expected fingerprints. `payloadtool` compares generated fingerprints against the expected values and exits with a non-zero code if they differ.

  The verify file format is one line per generator:
  ```
  my-generator: <sha256> entropy=<bits>
  ```
</ParamField>

<ParamField path="-m, --memory-stats" type="boolean">
  Print memory allocation statistics to stderr after generation completes. Reports total allocations, total deallocations, total bytes allocated, and peak bytes live.
</ParamField>

<ParamField path="--dump" type="string">
  Write the raw generated payload bytes to the specified file. Useful for visually inspecting what lading will send to the target.
</ParamField>

## Examples

### Validate a config parses and generates correctly

```bash
payloadtool lading.yaml
```

If the config is valid and payload generation succeeds, `payloadtool` exits with code 0 and logs how many bytes were generated:

```
INFO payloadtool: Loaded configuration, found 2 generators
INFO payloadtool: Payload generation took 1.2s
INFO payloadtool: Generator succeeded in generating 256 MiB of data
```

### Generate a fingerprint

```bash
payloadtool --fingerprint lading.yaml
```

Output:

```
my-generator: a3f8c2d1e4b6... entropy=7.9821
```

Store this output in a file and use `--verify` to detect regressions:

```bash
payloadtool --fingerprint lading.yaml > expected-fingerprints.txt
```

### Verify against expected fingerprints

```bash
payloadtool --fingerprint --verify expected-fingerprints.txt lading.yaml
```

On success:

```
INFO payloadtool: ✓ Fingerprint matches expected value
```

On mismatch, `payloadtool` logs the expected and actual fingerprints and exits with code 1.

### Run a single generator

```bash
payloadtool --generator-id my-http-generator --fingerprint lading.yaml
```

### Measure memory allocation

```bash
payloadtool --memory-stats lading.yaml
```

Outputs to stderr after generation:

```
Memory Statistics:
  Allocations:     12453
  Deallocations:   12218
  Total allocated: 1.23 GiB
  Peak live:       267.41 MiB
```

Use this to size the memory limits on your lading container.

### Dump generated payload to a file

```bash
payloadtool --dump ./payload-output.bin lading.yaml
```

The raw bytes that lading would send to the target are written to `payload-output.bin`. Inspect them with any binary or text tool to verify the payload matches expectations.

## Generator support

Not all generators support fingerprinting. The following generators do not currently support the `--fingerprint` flag:

- `SplunkHec`
- `FileTree`
- `ProcessTree`
- `ProcFs`
- `Container`
- `Kubernetes`

If you pass `--fingerprint` with one of these generators, `payloadtool` emits a warning and skips that generator.

## Fingerprint format

A fingerprint consists of two parts:

```
<sha256-hex> entropy=<shannon-entropy>
```

- **sha256-hex**: The SHA-256 hash of all generated payload bytes, encoded as lowercase hex.
- **entropy**: The Shannon entropy of the generated bytes in bits per byte (range: 0.0 to 8.0). Higher values indicate more random-looking data.

Two fingerprints match if both the hash is identical and the entropy values are within 0.01 bits.

## Log verbosity

Set `RUST_LOG` to control log output:

```bash
RUST_LOG=debug payloadtool lading.yaml
```
